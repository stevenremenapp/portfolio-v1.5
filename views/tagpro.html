TAGPRO: A CASE STUDY

INTRODUCTION

I worked on this project as the only frontend developer on the team, which initially included a backend developer and project architect, and later included a designer and quality assurance tester. I also unexpectedly worked as the initial designer for the application because the original designer didn't deliver what was expected. We were hired by the client to create their vision of a "Home Servicing Social Networking Platform." With a background as an owner of a successful custom cabinet business, the client was frustrated in many ways with sites like Thumbtack, Home Advisor, and Angie's List, and also wanted to add an element of Pinterest-like inspiration seeking to the site. It is from this point that we started working with the client to flesh out their ideas into a project they could launch to both home service professionals and consumers seeking home professionals. The project is set for beta release to the initial home professionals soon.

TOOLS

We are using Vue for the frontend, Go for the backend, Firebase for authentication, and Firestore for data storage, all set on the Google App Engine environment. Specifically, my work on the frontend leverages Vue, Vuex, Vue Router, BootstrapVue, and Firebase Authentication to deliver the TagPro single page application. I handled deploying to our dev, QA, and production environments on Google App Engine using the GCloud command line tool.

DEEP DIVE

I'm diving into a few of the more interesting parts of this project below; continue reading to learn more.

Design

The plan was to start out with a designer providing wireframes from the beginning, but it didn't happen and development proceeded. I ended up stepping in as the initial UI designer for the application, with direction from the client that they liked the Pinterest gridded layout, but they preferred it with sharper edges. I started out designing the homepage and public pages first, and will detail that evolution below.

Search

From the beginning we knew that search would be an integral part of this application, and we followed Thumbtack's lead of matching pros and projects to specific service type keywords. The project architect made the decision that GeoJSON search results wasn't necessary at this point in the project, so we designed search around a specific dictionary of service keywords and a limited set of filters that differed based on whether the user searched pros or projects. I matched the search results grid to the homepage grid and paginated to 30 results shown at once. When the user clicks "View More Results" another 30 result cards are added to the bottom of the grid, until the end is reached. I also implemented stateful search based on the queries in the URL so that if the user decided to press forward or back in their browser, or they were sent a search link URL, it would return the appropriate results.

Perhaps the most interesting part of the redesign for me was the inclusion of a custom search component that autocompleted the user's search input and allowed them to choose a search keyword for either a pro or project search. I used fuse.js, rather than explicitly matching strings to give the user some leeway with spelling their inputs. Similar to Thumbtack's implementation, without choosing a matching keyword the user could not complete a search. I built this component from scratch, and used Trevor Eyre's Vue autocomplete component to handle the autocomplete dropdown functionality. I learned a lot in building out this component, and it helped me refactor the original separate search inputs into one portable search component.

Tagging

Tagging (or favoriting) pros, projects, and photos was always part of the client's vision from the beginning. We created a set of tagging APIs that I can call to in order to take the correct actions for the user. I was initially using an ugly plus symbol that changed color based on tag status, but the designer moved toward badges in the banner elements and a badge (un-tagged) and circle indicator (tagged) in card elements to indicate tag status. My initial implementation of the redesign also waited for the API call to be returned before switching the UI, but the designer said it should happen immediately, so I modified the API calls to switch immediately and only return to the original state if the call could not be completed.

The image below shows the public professional profile page with a variety of tag indicators in the banner and card elements. The untagged badge in the card elements only shows on hover for large screens and is always present on smaller screens.

Dashboard & Public Pages

The dashboard was a good challenge for me because professionals and consumers have different priorities within the application and it had to flexible enough to share as many components as possible while also allowing differing functionality. For example, consumer users' main functionality is around tagging items and tracking the professionals they've contacted, while professional users are allowed to create a business profile, create projects, have their projects rated, and more. The organization of each type of users' dashboard reflects these priorities while maintaining a consistent UI. I also gave some care to considering how users unfamiliar with the application (read: everyone, because it's new) would require some instruction around what certain actions are, and when they can do them (professionals must complete their business profile before creating projects). I wish I had more time to dedicate to this aspect of the application, but I accomplished this by using permanent and non-intrusive instructional text combined with state-based alerts indicating required actions (particularly for professional users).

The public pages (i.e. the pro's business profile page and any project pages for the project's they publish) aggregate a public-friendly set of the data input by the pro in their dashboard related to their profile and projects. The goal was to make them attractive and photo-friendly so that professionals might share them as a portfolio. Users can find these pages by browsing the mainpage or by matching search keywords.

PROCESS

Initial Mock Up of TagPro

We started with the understanding that the client wanted to make a Thumbtack-like site that also incorporated the look, feel, and basic "pin" functionality of Pinterest. The client also sent along sites which had a look and feel that they liked. From this I started mocking out ideas using pencil and paper, and exploring pre-made Vue theme templates. Soon we realized that this would be a custom front-end and I started building out the initial version of various pages using mocked API data. From the beginning I built the design around Pinterest-like grids where possible. Soon after I added the beginning of the login system using Firebase and deployed it to the Google App Engine environment using Go functions to return mock data.

Feature Development

Feature development typically followed a standard pattern:

  1. Solidify feature requirements with the client.
  2. Collaborate with the backend developer and project architect to flesh out how to achieve the feature. Agree on a general direction for the feature.
  3. Collaborate with the backend developer to create a design document which detailed the API endpoints, data structure and storage, basic UI requirements and mock ups, and actions of the frontend that would use the APIs.
  4. Once the design document was approved by the project architect, we set to building out the feature based on those requirements.
  5. Once we had enough built out to try out the feature's functionality, the backend developer and I would collaborate on a deployment to our dev environment. We would test it out and iterate on fixes as needed, until it was functioning as expected.
  6. Move on, but return to iterate on the feature as necessary. Sometimes we would need to update past features in order to accommodate new features or unforeseen issues resulting from our initial feature design.

Client Meetings

The project was malleable from the beginning because the client's vision continued to change as we built out the project. As the frontend developer and initial designer I had frequent meetings with the client, especially at the beginning of the project as I was working to establish the overall look of the application. The client was frequently involved in testing the application, especially initially, and also helped manage a Trello task board. Client meetings were spaced further out as we all got more settled with the project, but they continued to be a fairly regular part of our iterative cycle. In addition to meetings, I also sent out release notes to the client to update them on the changes we made with each version deployment.

Final Redesign

I am not a designer. Shocking, I know. I was open with everyone involved in the project about this from the beginning. So with about two months left on the project, the client contracted a talented designer to redesign the site. After a few weeks, files started coming through in a Zeplin workspace and I could get to work. This meant a complete redesign of the design system I had set up, to take place over the timespan of a month. It was stressful at times, and involved creating new components (such as the custom search component), but it was definitely worth it because the application looks better and is more intuitive.

QA Testing

A QA tester joined the team in the last few months of the project as we approached the beta release. I worked with him to catch him up on the frontend of the app, and then continued to collaborate with him as he tested the application. He used our Trello task board to create cards for the issues he found, and then those were prioritized by the project architect and developers. Once prioritized, I worked to fix the discovered bugs and release updated versions of the application, along with the backend developer.

CONCLUSION

This was a challenging and rewarding project to work on. From learning the basics of Figma to handle the initial design, to collaborating with the backend developer and the project architect to create design documents and APIs for each feature, to holding regular meetings with the client to make sure the application is achieving what they want it to, to working with a QA tester to refine the project and fix bugs as possible, to moving toward beta release, we have been able to take this project from idea to implemented application on time and on budget. There is certainly room for improvement and expansion, but for now we will see how the release goes.
